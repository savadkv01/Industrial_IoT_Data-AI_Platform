# Implementation Plan

A phased roadmap to build the Industrial IoT Data & AI Platform. Each phase has a goal,
key deliverables, the matching agent skill, and an exit checklist. Build incrementally —
bring up only the Docker services a phase needs.

> Source spec: [system-design.md](../system-design.md) · Architecture: [architecture.md](architecture/architecture.md)
> Data model design (BI & AI): [DATA_MODELING.md](DATA_MODELING.md)

---

## Phase 1 — System Design (foundation)
**Goal:** Establish requirements, data flow, batch/stream split, and SLAs.
- Deliverables: [architecture.md](architecture/architecture.md), this plan.
- Exit: SLA targets quantified; every component mapped to a container.

## Phase 2 — Data Generation
**Goal:** Synthetic telemetry generator with realistic noise and failure simulation.
- Skill: `iot-telemetry-generator`
- Deliverables: `data-generator/` package, Kafka + file sinks, Pydantic schema.
- Exit: 500 machines stream to Kafka at target rate; failures labelled.

## Phase 3 — Streaming Pipeline
**Goal:** Kafka → Spark Structured Streaming into Bronze with exactly-once.
- Skill: `streaming-pipeline`
- Services: `kafka`, `minio`.
- Exit: Bronze Delta table grows continuously; checkpoints recover cleanly.

## Phase 4 — Lakehouse Design
**Goal:** Bronze → Silver → Gold medallion with DQ and schema evolution.
- Skill: `lakehouse-medallion`
- Services: `minio`, `trino` (optional), `duckdb`.
- Exit: Gold tables queryable via Trino/DuckDB; DQ checks pass.

## Phase 5 — Feature Engineering
**Goal:** Window/lag/rolling features + Feast feature store.
- Skill: `feature-store-feast`
- Exit: Features materialized online; point-in-time correctness verified.

## Phase 6 — ML Model Building
**Goal:** Predictive maintenance, anomaly detection, battery health models.
- Skill: `mlops-pipeline`
- Deliverables: `ml/` package — `predictive_maintenance`, `anomaly_detection`,
  `battery_health` training pipelines + `common` (time-aware splits, metrics, MLflow).
- Exit: Baseline metrics logged (AUC/F1/RMSE) with no time leakage. ✅

## Phase 7 — ML Pipeline (MLOps)
**Goal:** MLflow tracking + registry, retraining, batch vs real-time inference.
- Skill: `mlops-pipeline`
- Services: `mlflow`, `airflow`.
- Deliverables: `ml/pipeline.py` (train→log→register→promote), `ml/common/registry.py`
  (alias-based gated promotion), `ml/common/tasks.py` (shared task specs),
  `ml/inference/batch.py` (batch scoring from the registry), `orchestration/dags/train_models.py`
  (daily retrain DAG), `mlflow` + `airflow` docker services.
- Exit: Models versioned/staged in registry; retrain DAG runs. ✅

## Phase 8 — Real-time AI
**Goal:** Streaming anomaly detection: Kafka → Spark → model → alert.
- Skill: `streaming-pipeline`, `mlops-pipeline`
- Services: `kafka`, `minio`, `mlflow`.
- Deliverables: `streaming/spark/anomaly_alerts.py` (Structured Streaming job that scores
  `iot.telemetry` micro-batches with the aliased MLflow model and emits to `iot.alerts` +
  a Delta alert table), `streaming/spark/submit_alerts.sh`, `spark-alerting` compose service,
  per-machine cooldown throttling, and `AnomalyModelCache` for low-latency model hot-swaps.
- Exit: Anomalies flagged in near real-time (~20s end-to-end) and emitted to the `iot.alerts`
  topic + `s3a://lakehouse/silver/alerts/anomaly_alerts` Delta table. ✅

## Phase 9 — Model Serving
**Goal:** FastAPI inference service with low-latency model loading.
- Skill: `model-serving-fastapi`
- Services: `serving`, `mlflow`.
- Deliverables: `serving/app/` FastAPI service (`main.py`, `models.py`, `schemas.py`,
  `metrics.py`, `config.py`) that warm-loads the aliased MLflow models into an in-memory
  `ModelCache`, serves `POST /predict/{maintenance,battery,anomaly}` (echoing model
  name/alias/version), `GET /health`, `POST /reload`, and Prometheus `GET /metrics`;
  `serving/Dockerfile` (repo-root build context bundling the shared `ml` package) and a
  real `serving` compose service on port 8000.
- Exit: `/predict` endpoints serve cached models (no per-request load); OpenAPI docs at
  `/docs`; malformed payloads rejected with 422; every prediction carries its serving
  model version. ✅

## Phase 10 — Monitoring
**Goal:** System + data + model drift monitoring.
- Skill: `platform-monitoring`
- Services: `prometheus`, `grafana`.
- Exit: Dashboards live; drift reports generated by Evidently.

## Phase 11 — System Design Interviews
**Goal:** Use the platform to rehearse FAANG-style system design.
- Deliverables: `docs/interview/` scenarios (Uber/Tesla/Databricks style).

## Phase 12 — Production Incidents
**Goal:** Simulate and debug Kafka lag, Spark OOM, bad predictions, data skew, schema breaks.
- Deliverables: `docs/incidents/` runbooks.

---

## Suggested order of execution
1. Phases 1–4 establish the data backbone (build first).
2. Phases 5–7 add ML/MLOps on top of Gold.
3. Phases 8–10 add real-time + serving + observability.
4. Phases 11–12 are interview/ops rehearsal using the running platform.

## Engineering conventions
- One Python package per domain; promote notebook logic into packages.
- All long-running logic logs metrics for Prometheus.
- Config via `.env` (never commit secrets — see `.env.example`).
- Use a project **virtual environment** (`.venv`) for all Python work; never install into the system interpreter.
- Lint with `ruff`, test with `pytest` before each commit.
- Track behavioural acceptance per phase in [TEST_CASES.md](TEST_CASES.md) (separate from unit/syntax tests).
